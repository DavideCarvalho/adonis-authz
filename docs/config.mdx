`config/authz.ts` uses `defineConfig` plus the `stores` factory. Each store is a
lazy thunk â€” the `lucid` store imports `@adonisjs/lucid` only when selected, so
the peer dependency stays optional.

```ts title="config/authz.ts"
import { defineConfig, stores } from '@adonis-agora/authz'

export default defineConfig({
  default: 'lucid',
  stores: {
    memory: stores.memory(),
    lucid: stores.lucid({ /* connection, tables, autoCreateSchema */ }),
  },
})
```

## Stores

<Tabs items={['Lucid', 'Memory']}>
<Tab value="Lucid">

Persists RBAC in your database via `@adonisjs/lucid`.

```ts
stores.lucid({
  connection: 'pg',        // defaults to the app's default connection
  autoCreateSchema: false, // recommended: use the published migration instead
  tables: {                // optional name overrides
    roles: 'authz_roles',
  },
})
```

With `autoCreateSchema` left on (the default), the store runs
`CREATE TABLE IF NOT EXISTS` on first use. Set it to `false` and run the
published migration in production so schema changes are explicit.

</Tab>
<Tab value="Memory">

A zero-dependency, in-process store. Identical semantics to the Lucid store
(idempotency, tenant visibility, direct-grant rules), ideal for tests and tiny
apps. State is lost on restart.

```ts
stores.memory()
```

</Tab>

## Super-admin hook

Return `true` to allow, `false` to deny, or `undefined` to fall through to the
normal RBAC resolution. It receives the resolved `{ type, id }` reference.

```ts
defineConfig({
  // ...
  superAdmin: (user) => user.id === '1',
})
```

<Callout type="warn">
  The super-admin hook is the only hook whose `false` actively **denies** â€”
  it short-circuits even when a matching grant exists.
</Callout>

## Mapping a user to a reference

By default `user.id` (and optional `user.type`) is used. Override to fit your
model:

```ts
defineConfig({
  resolveUserRef: (user) => ({ type: 'account', id: user.uid }),
})
```

## Tenant resolver

Return a tenant id string (or `{ tenantId }`) for the current request; omit it
for single-tenant apps.

```ts
import { HttpContext } from '@adonisjs/core/http'

defineConfig({
  tenant: () => HttpContext.getOrFail().request.header('x-tenant'),
})
```

## Effective roles: `resolveRoles` & `roleGrants`

A user's **effective roles** are the union of three sources â€” and two config
seams let you feed them without seeding the authz store:

```
effective roles  =  context/global roles (token claim, via authkit)
                  âˆª  app roles      (the `resolveRoles` seam)
                  âˆª  store roles    (assigned via store.assignRole)
```

Every decision â€” `can`, `hasRole`, `hasAnyRole`, the
[route middleware](/docs/authz/middleware), a
[query scope](/docs/authz/query-scopes) and the
[frontend share](/docs/authz/react) â€” resolves against this same union, so they
never disagree.

### `resolveRoles` â€” app roles

Roles that live in **your** domain (typically a `user_roles`-style table), not in
the token and not in the authz store. Return them and they enter the union and
get mapped by `roleGrants`, exactly like global roles. Optional â€” omit it and
only the token and store decide.

```ts
defineConfig({
  resolveRoles: async (user, scope) => {
    const rows = await UserRole.query()
      .where('user_id', user.id)
      .if(scope, (q) => q.where('tenant_id', scope!.tenantId))
    return rows.map((r) => r.role)
  },
})
```

The callback receives the resolved `{ type, id }` reference and the active
`TenantScope`, so app roles can themselves be tenant-scoped.

### `roleGrants` â€” role â†’ permissions, no seed

A `role â†’ permissions/wildcards` map applied to the user's **effective** roles at
check time, without seeding the store. Union it into permission checks to grant a
role its abilities purely from config:

```ts
defineConfig({
  roleGrants: {
    editor: ['posts.*', 'comments.moderate'],
    auditor: ['audit.*'],
  },
})
```

Now a user with the `editor` role (from any of the three sources) passes
`can('posts.edit')` even though nothing was written to `authz_role_permission`.

<Callout type="info">
  `roleGrants` was formerly named `globalRoleGrants`. The provider reads **only**
  `roleGrants` now; an old `globalRoleGrants` key is silently ignored. It was
  renamed because it applies to *all* effective roles, not just global ones.
</Callout>

## Standalone schema helpers

With `autoCreateSchema: false` (recommended for production) the store no longer
creates its tables â€” you own the schema through a Lucid migration. The package
exports the exact same DDL the store would otherwise run, so the two paths never
drift:

```ts title="database/migrations/xxxx_create_authz_tables.ts"
import { BaseSchema } from '@adonisjs/lucid/schema'
import { createAuthzTables, dropAuthzTables } from '@adonis-agora/authz'

export default class extends BaseSchema {
  async up() {
    this.defer((db) => createAuthzTables(db))
  }

  async down() {
    this.defer((db) => dropAuthzTables(db))
  }
}
```

- `createAuthzTables(db, { tables? })` â€” idempotent (`CREATE TABLE IF NOT EXISTS`),
  dialect-aware (Postgres / MySQL / SQLite). Safe to call from a migration `up()`.
- `dropAuthzTables(db, { tables? })` â€” idempotent (`DROP TABLE IF EXISTS`),
  child-first, for the migration `down()`.
- `AUTHZ_TABLES` â€” the default table names (`authz_roles`, `authz_permissions`,
  `authz_role_permission`, `authz_user_role`, `authz_user_permission`).

Pass matching `tables` overrides to **both** the store config and these helpers so
they agree:

```ts
import { AUTHZ_TABLES } from '@adonis-agora/authz'

const tables = { ...AUTHZ_TABLES, roles: 'rbac_roles' }

// config/authz.ts â†’ stores.lucid({ autoCreateSchema: false, tables })
// migration       â†’ createAuthzTables(db, { tables })
```

<Callout type="info">
  `this.defer((db) => â€¦)` hands the helpers a Lucid query client whose dialect is
  read directly, so the `created_at` column type is chosen correctly
  (`TIMESTAMP` on Postgres, `DATETIME` elsewhere) even inside a migration.
</Callout>

## Sync catalog

Declare roles â†’ permissions to seed with `node ace authz:sync`:

```ts
defineConfig({
  catalog: {
    permissions: ['system.manage'],
    roles: {
      editor: ['posts.*'],
      viewer: ['posts.view', 'comments.view'],
    },
  },
})
```
